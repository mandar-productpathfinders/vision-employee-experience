## Section 1: What is a Microservice?

A microservice is a small, independently deployable service that does **one thing well**.

In Vision Enterprise UC2, each service owns a single domain:

| Service | Port | Its Responsibility |
|---------|------|-------------------|
| `hrms-api` | 8101 | Employee profiles, life-events, pending tasks, task completion |
| `onboarding-api` | 8102 | IT onboarding form, HR profile form |
| `access-api` | 8103 | GitHub, Slack, and additional tool access requests |
| `documents-api` | 8104 | Contract, code of conduct, compliance training acceptances |
| `calendar-api` | 8105 | Connect meeting bookings (manager intro, buddy, townhall) |
| `admin-api` | 8106 | The prompt registry and the tool registry (this is the keystone) |
| `announcement-api` | 8107 | The runtime-added tool's target — backs the announcement queue |

**Why split them up?**

- Each service is small enough to read in one sitting. You can hand `documents-api` to a fresh joiner on day one and they can extend it without learning the rest of the system.
- Services can be scaled independently. If meeting bookings spike during a hiring season, only `calendar-api` needs more capacity.
- Teams can own individual services without stepping on each other.
- A failure in `announcement-api` does not take down the onboarding flow.

Each service has:
- Its own `Dockerfile`
- Its own Python dependencies (FastAPI + psycopg + a few common libraries)
- Its own `/health` endpoint
- A Swagger UI at `/docs` (auto-generated by FastAPI)

## Section 4: What is MCP?

### The Problem

Claude (the AI model) can reason, write, and plan — but it cannot access your databases, call your APIs, or retrieve live data on its own. It only has access to what you put in its input.

One option: put everything in the prompt. But every employee profile, the full task catalogue, the entire tool list, would be tens of thousands of tokens per turn — slow, expensive, and changing constantly.

Better option: give Claude **tools** it can call on demand.

### What MCP Provides

MCP (Model Context Protocol) is a standard way to define and serve tools to an AI model. The MCP server in this project acts as a **gateway** between the agent and your microservices:

```
Agent thinks: "I need this employee's profile to figure out what to help them with."
     │
     │ tool_use block: { name: "get_employee_profile", input: { employee_id: "E1001" } }
     ▼
MCP Server receives the request
     │
     │ Looks up "get_employee_profile" in its in-memory registry
     │ Finds endpoint_url = "http://hrms-api:8101/profile", method = "GET"
     │ Dispatches: GET http://hrms-api:8101/profile?employee_id=E1001
     ▼
hrms-api queries PostgreSQL (or Redis if cached)
     │
     ▼
MCP Server returns result to the agent
     │
Agent thinks: "Arjun is in JUST_JOINED. Nine tasks pending. Let me ask what he wants to do."
```

**Analogy:** The agent is a chef who cannot leave the kitchen. The MCP server is the restaurant manager who takes orders, fetches what is needed from the warehouse (microservices), and delivers the results back.

## Section 5: The Dynamic Tool Registry — the UC2 Twist

This is where UC2 differs from typical MCP implementations and from Use Case 1.

In Use Case 1, the MCP server has a `TOOL_REGISTRY` array hardcoded in `server.js`. Adding a tool requires editing the file, committing, and redeploying.

In Use Case 2, **the MCP server has no hardcoded tools**. Open `mcp-server/index.js` and look at the top of the file:

```javascript
let toolRegistry = []; // array of tool definitions from admin-api

async function reloadRegistry() {
  const { data } = await axios.get(`${ADMIN_API_URL}/tools`, {
    params: { active_only: true },
    timeout: 3000,
  });
  toolRegistry = (data || []).map(normalise);
  lastReloadAt = new Date().toISOString();
}

// At startup and every 5 seconds:
setInterval(reloadRegistry, RELOAD_MS);  // RELOAD_MS = 5000
```

The registry starts empty. The MCP server calls `admin-api:8106/tools` on startup and every five seconds thereafter. Whatever rows it finds in `admin_tools` (with `is_active = TRUE`) become the live tool catalogue.

This is the line that makes the demo possible:

```javascript
setInterval(reloadRegistry, RELOAD_MS);
```

That single line is the difference between an agent platform you can grow and a workflow tool you have to redeploy.

### Why a Poll Instead of a Push?

Five-second polling is the simplest possible mechanism. The trade-off is up to five seconds of lag between an admin saving a tool and the agent seeing it. Acceptable for this use case; not acceptable for everything. If you needed sub-second propagation you would:
- Replace the poll with a webhook from admin-api to the MCP server
- Or use Postgres `LISTEN/NOTIFY` plus a long-poll
- Or use a message bus (Redis pub/sub, NATS) in front of the registry

The poll loop is also pessimistic: if admin-api is down, the MCP server keeps serving whatever registry it last successfully loaded. This is the right failure mode — better stale capabilities than no capabilities.

**`GET /health`** — Health and current registry size:

```bash
curl -s http://localhost:8100/health | python3 -m json.tool
```

Expected:
```json
{
  "status": "ok",
  "service": "mcp-server",
  "tools_loaded": 9,
  "last_reload_at": "2026-...",
  "last_reload_error": null
}
```

Nine tools. Same number as Lab 1's seeded `admin_tools`. The registry was rebuilt on startup from the database.

**`GET /tools`** — The full registry as the agent would see it:

```bash
curl -s http://localhost:8100/tools | python3 -m json.tool | head -40
```

You will see a `tools` array. Each entry has `name`, `description`, `input_schema`, and a `_meta` block with the dispatch endpoint. The agent only reads `name`, `description`, and `input_schema` — the `_meta` is for the MCP server's own use.

Optionally filter by event:
```bash
curl -s "http://localhost:8100/tools?event_code=JUST_JOINED" | python3 -m json.tool | head -40
```

The filter returns tools scoped to JUST_JOINED **or** tools with no event scope (the three global tools — get_employee_profile, list_employee_events, mark_task_complete — are available everywhere).

**`POST /tools/call`** — Dispatch a tool. This is how the agent invokes tools:

```bash
curl -s -X POST http://localhost:8100/tools/call \
  -H "Content-Type: application/json" \
  -d '{"name": "get_employee_profile", "input": {"employee_id": "E1001"}}' \
  | python3 -m json.tool
```

The response is wrapped:
```json
{
  "ok": true,
  "tool": "get_employee_profile",
  "latency_ms": 8,
  "result": { ...Arjun's profile... }
}
```

**`POST /tools/reload`** — Force an immediate registry refresh:

```bash
curl -s -X POST http://localhost:8100/tools/reload | python3 -m json.tool
```

The Admin Console calls this endpoint after a save so changes are visible without waiting for the next poll. You will see it in action in Section 8.

## Section 7: Tracing a Tool Call Through the System

Let us trace `book_connect_meeting` end-to-end. Pick a tool that writes data so you can observe the result in the database.

**Step 1 — Call the MCP server:**

```bash
curl -s -X POST http://localhost:8100/tools/call \
  -H "Content-Type: application/json" \
  -d '{
    "name": "book_connect_meeting",
    "input": {
      "employee_id": "E1001",
      "meeting_type": "MANAGER_INTRO",
      "preferred_dates": ["2026-05-20","2026-05-21","2026-05-22"]
    }
  }' | python3 -m json.tool
```

**Step 2 — The MCP server's dispatch logic** (in `mcp-server/index.js`):

```javascript
const tool = toolRegistry.find((t) => t.name === name);
// tool._meta.endpoint = "http://calendar-api:8105/book"
// tool._meta.method = "POST"

response = await axios({
  method: "POST",
  url: "http://calendar-api:8105/book",
  data: input,
  timeout: 10000,
});
```

The MCP server does not know what `book_connect_meeting` does. It only knows the endpoint to call and the method to use. The mapping comes entirely from the row in `admin_tools`.

**Step 3 — calendar-api handles it:**

It writes to `connect_meetings` (status `booked`, a chosen slot, the manager's name), and updates `employee_task_status` for `JJ_MANAGER_INTRO` to `completed`.

**Step 4 — Verify the result:**

```bash
docker exec -it ve-uc2-postgres psql -U ve_user -d vision_uc2 \
  -c "SELECT employee_id, meeting_type, status, booked_with, booked_slot FROM connect_meetings ORDER BY id DESC LIMIT 1;"
```

You should see a row for E1001, MANAGER_INTRO, status=booked, booked_with=Priya Nair, with the chosen slot.

```bash
docker exec -it ve-uc2-postgres psql -U ve_user -d vision_uc2 \
  -c "SELECT status FROM employee_task_status WHERE employee_id='E1001' AND task_code='JJ_MANAGER_INTRO';"
```

You should see `completed`. The task tracker updated automatically.

**Total round-trip:** approximately 20–40 ms. No ML model is involved. The MCP server's value here is not performance — it is **uniformity**. From the agent's perspective, `book_connect_meeting` is exactly the same shape as `get_employee_profile` or any future tool you add. One call shape, one result shape, regardless of what is behind each tool.

## Section 8: Adding a Tool LIVE — the Demo Climax

This section is the centrepiece of the use case. You will add a new capability to the agent without writing any code, without restarting any service, and without redeploying anything.

### The Setup

The `announcement-api` microservice already exists (port 8107). It exposes `POST /send` which writes to `announcement_queue`. The schema supports it, the service runs, and yet the agent cannot use it — because there is no row in `admin_tools` for it. The agent's capabilities are defined by the table, not by what is reachable on the network.

### The Add — via admin-api

You have two options. The clean way for this lab is to use the admin-api directly. (In Lab 3 you do the same thing through the Admin Console UI.)

```bash
curl -s -X POST http://localhost:8106/tools \
  -H "Content-Type: application/json" \
  -d '{
    "tool_name": "send_joiner_announcement",
    "display_name": "Send Joiner Announcement",
    "description": "Queue an announcement to the team about a new joiner. Use this near the end of onboarding once the employee has completed most tasks. Always confirm with the employee before calling. recipient_list is a comma-separated CSV of email addresses or distribution lists.",
    "input_schema": {
      "type": "object",
      "properties": {
        "employee_id":    {"type": "string"},
        "recipient_list": {"type": "string"},
        "subject":        {"type": "string"},
        "body":           {"type": "string"}
      },
      "required": ["employee_id","recipient_list","subject","body"]
    },
    "endpoint_url": "http://announcement-api:8107/send",
    "http_method": "POST",
    "event_code": "JUST_JOINED",
    "linked_task_code": null
  }' | python3 -m json.tool
```

Expected response:
```json
{ "ok": true, "tool_name": "send_joiner_announcement", "updated_at": "2026-..." }
```

### Watch the MCP Server Pick It Up

Wait up to five seconds, then check the MCP server's view of its registry:

```bash
curl -s http://localhost:8100/health | python3 -m json.tool
```

Expected: `"tools_loaded": 10`.

You can also force an immediate reload instead of waiting:

```bash
curl -s -X POST http://localhost:8100/tools/reload | python3 -m json.tool
```

This is the call the Admin Console makes after a save in Lab 3.

### Use the New Tool — Through the MCP Server

```bash
curl -s -X POST http://localhost:8100/tools/call \
  -H "Content-Type: application/json" \
  -d '{
    "name": "send_joiner_announcement",
    "input": {
      "employee_id": "E1001",
      "recipient_list": "all-engineering@visionenterprise.com",
      "subject": "Welcome Arjun Kumar",
      "body": "Please join us in welcoming Arjun, our new Software Engineer in Bengaluru."
    }
  }' | python3 -m json.tool
```

Verify in the database:

```bash
docker exec -it ve-uc2-postgres psql -U ve_user -d vision_uc2 \
  -c "SELECT id, subject, status FROM announcement_queue ORDER BY id DESC LIMIT 1;"
```

You should see one queued row.

**Recap of what just happened:**
1. The announcement-api microservice was already running but unreachable to the agent.
2. You added one row to `admin_tools` (via admin-api).
3. The MCP server picked up the row from its next poll.
4. The capability is now part of the agent's tool list. In Lab 3 you will see the agent actually call it.

No code change. No redeploy. The capability surface of the agent moved from 9 to 10 in five seconds.

## Section 9: How the Agent Loads Its Prompt (Preview)

You will run this in Lab 3, but the mechanism is in scope for this lab. On every chat turn, the agent calls admin-api twice — once for the ROUTER prompt, once for the event-specific prompt — and concatenates them.

```bash
# This is exactly the call the agent makes every turn:
curl -s http://localhost:8106/prompt/ROUTER | python3 -m json.tool
curl -s http://localhost:8106/prompt/JUST_JOINED | python3 -m json.tool
```

Two separate API calls, two independently editable prompts, one concatenated system prompt sent to the model. The agent code (`agent/index.js`) is fewer than 200 lines because most of the behaviour is in these two rows.

If you edited either prompt right now — for example via `PUT /prompt/JUST_JOINED` — the **next** chat turn would reflect it. There is no caching. The prompt is loaded fresh every turn. This is intentional: the cost of one tiny HTTP call is negligible compared to the value of always having the current policy.

## Troubleshooting

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| `mcp-server` `tools_loaded: 0` | admin-api was not ready when MCP server first polled | Wait 5 seconds — the next poll will succeed; or `curl -X POST http://localhost:8100/tools/reload` |
| MCP returns `{"ok": false, "error": "tool '<name>' not found"}` | Tool not in active registry, or you added it but did not wait for the poll | Run `curl -X POST http://localhost:8100/tools/reload` to force reload |
| MCP returns timeout error | Target microservice slow or down | `docker compose ps` and check logs for the target service |
| Swagger UI not loading at `/docs` | Service not started | Run `docker compose up -d <service-name>` |
| `connect_meetings` row missing after T2.11 | Tool call failed silently before reaching the service | Inspect the MCP response — `ok` should be true; if false, read `error` |
| 409 conflict when adding the announcement tool in T2.13 | You already added it (or ran the lab twice) | `DELETE FROM admin_tools WHERE tool_name='send_joiner_announcement'` and re-run, OR skip T2.13 and continue |
